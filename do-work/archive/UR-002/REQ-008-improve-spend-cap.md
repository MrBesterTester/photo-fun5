# Improve Spend Cap with Token-Based Cost Estimation

## What
Upgrade the spend cap implementation (REQ-006) from a flat per-request estimate to token-based cost estimation using actual Gemini API response token counts, matching the approach used in samkirk-v3.

## Current State
- REQ-006 implemented a flat 5-cent-per-request estimate (`SPEND_PER_REQUEST_CENTS = 5`)
- No connection to actual Gemini API usage data
- No way to validate estimates against real costs

## Desired Improvements
- Use actual token counts from the Gemini API response (`usageMetadata`) to estimate cost
- Define per-token pricing constants (input tokens, output tokens) matching current Gemini pricing
- Apply a minimum cost floor per request (e.g., $0.001) so even tiny requests count
- Add a `estimateLlmCost(inputTokens, outputTokens)` function
- Fall back to a text-based token estimate if API doesn't return token counts
- Record spend using the calculated cost instead of the flat rate
- Reference: samkirk-v3's implementation in `web/src/lib/spend-cap.ts`

## Context
- samkirk-v3 uses Firestore transactions for atomic spend updates; photo-fun5 uses Redis `incrby` which is simpler but still adequate
- The key improvement is cost accuracy, not storage mechanism
- Gemini API responses include `usageMetadata` with `promptTokenCount` and `candidatesTokenCount` (or similar fields depending on SDK version)

## Exploration

**Key file:** `api/image-edit.ts` — single serverless endpoint handling everything

**Current spend cap code:**
- Constants at lines 59-74: `MONTHLY_SPEND_CAP_CENTS=2000`, `SPEND_PER_REQUEST_CENTS=5`, `SPEND_KEY_TTL_SECONDS=35 days`
- Separate Redis client (`spendRedis`) using Upstash credentials
- Spend check (lines 138-156): reads `spend:{YYYY-MM}` key, returns 503 if over cap, fail-open on error
- Spend increment (lines 269-282): `spendRedis.incrby(spendKey, SPEND_PER_REQUEST_CENTS)` after successful response, TTL set on first increment

**Gemini API call & response:**
- Call at lines 201-205: `ai.models.generateContent({model: 'gemini-3-pro-image-preview', contents, config})`
- Response includes `usageMetadata` with `promptTokenCount`, `candidatesTokenCount` — **not currently used**
- Only `response.candidates?.[0]` is accessed for image/text extraction

**Reference implementation (samkirk-v3):**
- `COST_PER_1K_INPUT_TOKENS_USD = 0.00125` ($1.25/1M input)
- `COST_PER_1K_OUTPUT_TOKENS_USD = 0.00375` ($3.75/1M output)
- `MIN_COST_PER_CALL_USD = 0.001` floor
- `estimateLlmCost(inputTokens, outputTokens)` function
- `estimateTokensFromText(text)` fallback (~4 chars per token)

**Key insight:** Redis `incrby` works with integers (cents). Need to convert token-based USD cost to cents and use `Math.ceil` to avoid underestimating. The minimum floor of $0.001 = 0.1 cents → rounds up to 1 cent minimum.
